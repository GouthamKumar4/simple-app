# MedCore -- Secure Patient Record System

## Requirement Clarification (Layman Version)

**Document Version:** 1.0\
**Audience:** Business + Non-Technical Stakeholders\
**Purpose:** Explain system requirements in simple language with visual
diagrams

------------------------------------------------------------------------

# 1. What Are We Building? (Simple Explanation)

We are building a secure online system where clinic staff can:

-   Add patient information\
-   Update patient details\
-   Search patient records\
-   Safely store medical-related data\
-   Control who can see or edit data

The system should be: - Easy to use\
- Secure\
- Reliable\
- Scalable for future growth

------------------------------------------------------------------------

# 2. Functional Requirements (In Simple Terms)

## 2.1 Patient Management

The system must allow staff to:

-   Create new patient records\
-   Edit patient details\
-   View patient information\
-   Soft delete (mark as inactive, not permanently remove)\
-   Search patients by name, phone, or ID

------------------------------------------------------------------------

## 2.2 User Roles (Who Can Do What)

  Role           What They Can Do
  -------------- ------------------------------------------------
  Admin          Full control (add, edit, delete, manage users)
  Doctor         View and update patient records
  Receptionist   Add new patients and view records

------------------------------------------------------------------------

## 2.3 Login & Security

-   Staff must log in using company email\
-   Multi-Factor Authentication (MFA) required\
-   Every user action must be recorded (audit log)

------------------------------------------------------------------------

# 3. How The System Works (Simple Flow)

``` mermaid
flowchart LR
    User --> Login
    Login --> Authentication
    Authentication --> API
    API --> Database
    API --> AuditLog
```

Explanation:

1.  User logs in\
2.  System verifies identity\
3.  User interacts with system\
4.  Data stored securely in database\
5.  All actions logged

------------------------------------------------------------------------

# 4. Multi-Clinic Support (Simple View)

Each clinic should only see its own patients.

``` mermaid
flowchart TD
    ClinicA --> System
    ClinicB --> System
    System --> Database
    Database --> ClinicDataIsolation
```

This ensures no clinic can access another clinic's data.

------------------------------------------------------------------------

# 5. Non-Functional Requirements (In Layman Terms)

These describe how well the system should perform.

------------------------------------------------------------------------

## 5.1 Speed

-   Pages should load in less than 2 seconds\
-   System should handle 60 users at same time

------------------------------------------------------------------------

## 5.2 Availability

-   System should be available 99.9% of the time\
-   Short maintenance windows allowed

------------------------------------------------------------------------

## 5.3 Security

-   All communication must be encrypted\
-   Patient data encrypted in storage\
-   No shared passwords\
-   Strict access control

------------------------------------------------------------------------

## 5.4 Backup & Recovery

-   Daily backups\
-   If system fails, recovery within 4 hours\
-   Data loss should not exceed 24 hours

------------------------------------------------------------------------

## 5.5 Monitoring

System should automatically:

-   Detect downtime\
-   Alert on unusual login activity\
-   Monitor performance issues

------------------------------------------------------------------------

# 6. Growth Expectations

Today: - 20 clinics\
- 250 users

Future: - 100+ clinics\
- 1,000+ users

System must scale smoothly.

------------------------------------------------------------------------

# 7. High-Level Architecture (Simple View)

``` mermaid
flowchart LR
    User --> WebApp
    WebApp --> AuthService
    WebApp --> Database
    WebApp --> Monitoring
```

Where:

-   WebApp = Main system\
-   AuthService = Login & MFA\
-   Database = Stores patient data\
-   Monitoring = Watches system health

------------------------------------------------------------------------

# 8. Summary

We are building a:

-   Secure\
-   Scalable\
-   Multi-clinic\
-   Role-based\
-   Auditable

patient record management system.

This document explains requirements in simple language so both business
