# MedCore Secure Patient System

## Data Design (Patient-Centric, Deep Explanation with Why + Mermaid Diagrams)

------------------------------------------------------------------------

# 1. Data Design in Context of a Real Patient Use Case

Let us start with a real scenario.

A patient named Ravi visits Clinic A for consultation.

The receptionist creates his record. The doctor updates his diagnosis.
Later, Ravi returns for follow-up. An admin audits record access.

Behind all these actions, the most important component working silently
is the **data layer**.

If data design is weak:

-   Records may be inconsistent.
-   Patient history may be lost.
-   Duplicate records may appear.
-   Cross-clinic data leakage may occur.
-   Performance may degrade as clinics grow.

Data design is not just about tables.

It is about correctness, integrity, traceability, scalability, and
compliance.

------------------------------------------------------------------------

# 2. Why We Choose a Relational Database

Patient data is highly structured.

Each patient has:

-   Name
-   Date of Birth
-   Phone Number
-   Medical ID
-   Clinic association
-   Audit history

These relationships must be:

-   Strictly validated
-   Consistent
-   Transaction-safe
-   Queryable with joins

Relational databases provide:

-   ACID transactions
-   Foreign key constraints
-   Structured schema enforcement
-   Strong consistency

Healthcare systems require correctness over flexibility.

If diagnosis is saved but audit log fails, the entire transaction must
roll back.

This is why relational design is selected.

# 4. Multi-Tenant Data Strategy

MedCore supports multiple clinics in one system.

We use:

Shared Database + Shared Schema + clinic_id column

Why?

-   Cost-efficient
-   Easy to manage for MVP
-   Simplifies deployment
-   Supports future scaling

Every patient record contains:

clinic_id

This ensures data isolation at query level.

Diagram:

``` mermaid
flowchart TD
    ClinicA -->|clinic_id=A| PatientRecords
    ClinicB -->|clinic_id=B| PatientRecords
```

Application enforces filtering by clinic_id.

Without this:

-   Clinic A could see Clinic B data.
-   Legal exposure increases.

# 5. Patient Table Design (Deep Explanation)

Patient table contains:

-   patient_id (Primary Key)
-   clinic_id (Foreign Key)
-   medical_id
-   first_name
-   last_name
-   phone
-   date_of_birth
-   is_deleted (Soft delete flag)
-   created_at
-   updated_at

Why soft delete?

Instead of permanently deleting a patient:

We mark is_deleted = true.

Why?

-   Compliance requirements
-   Audit traceability
-   Prevent accidental loss
-   Preserve historical integrity

------------------------------------------------------------------------

# 6. Referential Integrity

We enforce foreign keys:

-   Patient must belong to a valid clinic.
-   User must belong to a valid clinic.
-   Audit log must reference valid user and patient.

Diagram:

``` mermaid
flowchart LR
    Clinic --> Patient
    Clinic --> User
    User --> AuditLog
    Patient --> AuditLog
```

If referential integrity is not enforced:

-   Orphan records may appear.
-   Audit trail becomes unreliable.
-   Data inconsistencies increase.

------------------------------------------------------------------------

# 7. Transaction Design

When doctor updates diagnosis:

Steps:

1.  Update patient record.
2.  Insert audit log.
3.  Commit transaction.

Diagram:

``` mermaid
sequenceDiagram
    participant Doctor
    participant API
    participant Database

    Doctor->>API: Update Diagnosis
    API->>Database: Update Patient
    API->>Database: Insert Audit Log
    Database-->>API: Commit Transaction
```

If audit insert fails:

Entire transaction rolls back.

This preserves integrity.

------------------------------------------------------------------------

# 8. Indexing Strategy

Indexes are created on:

-   (clinic_id, medical_id)
-   (clinic_id, phone)
-   (clinic_id, last_name)

Why composite indexes?

Because all queries filter by clinic_id first.

Without proper indexing:

-   Full table scans occur.
-   Performance degrades.
-   Scalability suffers.

Indexes ensure fast search under 1000+ users.

------------------------------------------------------------------------

# 9. Data Security at Database Level

Security includes:

-   Encryption at rest
-   Encrypted backups
-   Private network access
-   No public exposure

Even if storage layer is compromised:

Data remains encrypted.

------------------------------------------------------------------------

# 10. Backup & Recovery Design

Backup Model:

``` mermaid
flowchart TD
    ProductionDB --> DailyFullBackup
    ProductionDB --> TransactionLogBackup
```

Why daily backup?

To protect against:

-   Accidental deletion
-   Data corruption
-   Cyber incidents

Restore capability ensures:

RPO ≤ 24 hours RTO ≤ 4 hours

Regular restore testing validates recovery readiness.

------------------------------------------------------------------------

# 11. Future Data Evolution Strategy

As system grows:

-   Introduce read replicas
-   Partition large tables
-   Separate reporting database
-   Possibly shard by clinic group

Current design allows evolution without redesign.
