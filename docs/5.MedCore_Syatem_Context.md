# MedCore Secure Patient Management System

## System Context & Boundary (Deep Explanation with Why + Mermaid Diagrams)

------------------------------------------------------------------------

# 1. What Is System Context? (Detailed Explanation)

System Context is the highest-level architectural view of a system, and
it defines the complete boundary between what we are responsible for
building and controlling versus what exists outside our responsibility
but still interacts with or affects our system.

In simple terms, System Context answers very important questions:

-   What exactly are we building?
-   Who uses it?
-   What external systems does it rely on?
-   Where does trust begin and end?
-   What do we control, and what do we not control?
-   Where could external failures impact us?

Without clearly defining system context, architecture decisions become
incorrect because teams may assume control over systems they do not own
or forget to protect boundaries that are exposed to external risk.

System Context is a **black-box view**, meaning we look at the system as
a whole without diving into internal components yet.

------------------------------------------------------------------------

# 2. System Under Design

The system under design is:

**MedCore Secure Patient Management Application**

This includes every component that is:

-   Developed by our engineering team
-   Deployed within our cloud environment
-   Managed under our operational processes
-   Secured under our security policies

Anything outside of this boundary is not under direct engineering
control.

------------------------------------------------------------------------

# 3. What Is Inside the System Boundary?

Everything inside the boundary is fully controlled by our organization.

Internal components include:

-   Web Application (Frontend UI)
-   Backend API
-   Business Logic Layer
-   Data Access Layer
-   Database
-   Audit Logging Module
-   Monitoring & Logging Configuration

These components:

-   Share internal trust relationships
-   Follow internal security rules
-   Are deployed within controlled infrastructure
-   Are protected by network isolation rules

``` mermaid
    end
```

Everything inside this subgraph is our responsibility.

------------------------------------------------------------------------

# 4. External Actors (Human Users)

External actors are real-world users who interact with the system but
are not part of it.

These include:

-   Doctor
-   Receptionist
-   Admin

They access the system via web browsers over the internet.

They are outside the system boundary because:

-   We do not control their devices.
-   We do not control their network security.
-   We do not control how they store passwords.
-   Their traffic originates from untrusted networks.

Diagram:

``` mermaid
flowchart LR
    Doctor --> Internet
    Receptionist --> Internet
    Admin --> Internet
    Internet --> MedCoreSystem
```

This creates the **primary trust boundary** between untrusted traffic
and trusted infrastructure.

------------------------------------------------------------------------

# 5. External Systems (Dependencies)

External systems are services that MedCore depends on but does not fully
control.

Examples:

-   Identity Provider (Authentication + MFA)
-   Email Service (if notifications exist)
-   SMS Gateway (if OTP required)
-   External Monitoring Tool (if SaaS-based)

These systems:

-   Operate independently
-   Have their own SLAs
-   May experience outages
-   May introduce latency
-   May change APIs over time

Diagram:

``` mermaid
flowchart LR
    MedCoreSystem --> IdentityProvider
    MedCoreSystem --> EmailService
```

These integrations must be secured and validated.

# 7. Trust Boundaries (Critical Concept)

A trust boundary is a location where data crosses from a less trusted
environment into a more trusted one.

Every time data crosses a boundary, validation and security enforcement
must occur.

------------------------------------------------------------------------

## 7.1 Primary Trust Boundary (Internet → Application)

Internet traffic is untrusted.

Reasons:

-   Attackers exist on the internet.
-   Requests can be manipulated.
-   Credentials can be stolen.
-   Traffic can be malicious.

Diagram:

``` mermaid
flowchart LR
    InternetUsers -->|Untrusted| TLSGateway
    TLSGateway -->|Validated| WebApp
```

At this boundary we enforce:

-   TLS encryption
-   Input validation
-   Authentication
-   Rate limiting
-   WAF policies (if implemented)

------------------------------------------------------------------------

## 7.2 Secondary Trust Boundary (Application → Identity Provider)

Identity Provider is external.

We must:

-   Validate token signatures
-   Validate issuer
-   Validate audience
-   Handle token expiration
-   Handle provider outages gracefully

If token validation fails, access must be denied.

------------------------------------------------------------------------

## 7.3 Internal Trust Boundary (API → Database)

Even inside our infrastructure, not everything is automatically trusted.

Database must:

-   Accept connections only from API layer
-   Not be publicly accessible
-   Use authentication and encryption

Diagram:

``` mermaid
flowchart LR
    API -->|Private Network| Database
```

This ensures database cannot be attacked directly from internet.

------------------------------------------------------------------------

# 8. Availability Dependencies in Context

System Context also reveals external availability dependencies.

Example:

If Identity Provider fails:

-   Users cannot authenticate.
-   System becomes partially unavailable.

This means:

System availability depends on external SLA.

Architecturally, this must be acknowledged.

------------------------------------------------------------------------

# 9. Security Exposure Points

System Context helps identify exposure points:

-   Public internet endpoint
-   Identity provider integration
-   External API integrations
-   Email/SMS callbacks

These are areas requiring extra validation and monitoring.

------------------------------------------------------------------------

# 10. System-Level vs Component-Level (Black Box vs White Box)

System-Level View = Black Box

We see:

Inputs → System → Outputs

Example:

Doctor request → MedCore → Patient data response

We do not see internal modules.

------------------------------------------------------------------------

Component-Level View = White Box

We see:

-   Controllers
-   Middleware
-   Services
-   Repositories
-   Database operations

This is deeper internal design.

Why this abstraction matters:

Executives care about:

-   Availability
-   Risk exposure
-   Compliance
-   External dependencies

Developers care about:

-   Internal module design
-   Code separation
-   Transaction handling

Platform engineers must understand both levels.

------------------------------------------------------------------------

# 11. Why System Context Must Be Defined First

If we skip system context:

-   We may expose database publicly.
-   We may forget external SLA dependency.
-   We may misunderstand security boundaries.
-   We may misconfigure network segmentation.
-   We may assume control over systems we do not own.

System Context ensures architectural clarity before infrastructure or
component design begins.

------------------------------------------------------------------------

# 12. Final Summary

System Context clearly defines:

-   What we own
-   What we do not own
-   Who interacts with us
-   Where trust boundaries exist
-   Where security must be enforced
-   Where external dependencies introduce risk

It is the foundation of secure architecture, resilient deployment, and
scalable design.

Without a clearly defined system boundary, architecture becomes fragile
and assumptions become dangerous.
