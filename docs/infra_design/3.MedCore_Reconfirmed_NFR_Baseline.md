# MedCore Secure Patient System

# Reconfirmed Non-Functional Requirements (NFR) Baseline

## Infrastructure Design Foundation Document

Document Version: 2.0\
Purpose: Reconfirm, refine, and formalize all Non-Functional
Requirements before Infrastructure Design execution.

------------------------------------------------------------------------

# 1. Why We Reconfirm NFRs

Before designing infrastructure, we must ensure that every
infrastructure decision directly supports approved Non-Functional
Requirements.

Non-Functional Requirements define:

-   How reliable the system must be
-   How fast it must respond
-   How secure it must operate
-   How recoverable it must be
-   How scalable it must become
-   How observable it must remain

Infrastructure Design exists to physically enforce these requirements.

------------------------------------------------------------------------

# 2. Availability Requirements

## 2.1 Business Expectation

The system must be available 99.9% of the time in production.

This means downtime must be limited and controlled.

## 2.2 Formalized Availability NFRs

  ID       Requirement
  -------- -----------------------------------------------
  NFR-A1   Production uptime must be ≥ 99.9% monthly
  NFR-A2   No single point of failure in compute layer
  NFR-A3   Maintenance must not exceed 4 hours per month

## 2.3 Infrastructure Implication

To satisfy 99.9% uptime:

-   Minimum two application instances
-   Load balancing
-   Health probes
-   Managed high-availability services

Diagram:

``` mermaid
flowchart LR
    Users --> LoadBalancer
    LoadBalancer --> AppInstance1
    LoadBalancer --> AppInstance2
```

If one instance fails, traffic is redirected automatically.

# 3. Performance Requirements

## 3.1 Business Expectation

-   Pages must load in under 2 seconds
-   System must support 60 concurrent users initially
-   Growth toward 1000+ users

## 3.2 Formalized Performance NFRs

  ID       Requirement
  -------- ------------------------------------------------
  NFR-P1   95% of API responses must be \< 2 seconds
  NFR-P2   Support minimum 60 concurrent active users
  NFR-P3   Performance must not degrade under normal load

## 3.3 Infrastructure Implication

-   Proper compute sizing
-   Monitoring of latency
-   Autoscaling configuration

Performance Flow:

``` mermaid
sequenceDiagram
    participant User
    participant API
    participant DB

    User->>API: Request
    API->>DB: Query
    DB-->>API: Result
    API-->>User: Response <2s
```

Latency must be monitored continuously.

# 4. Scalability Requirements

## 4.1 Business Expectation

System must grow from:

-   20 clinics → 100+ clinics
-   250 users → 1000+ users

## 4.2 Formalized Scalability NFRs

  ID       Requirement
  -------- ------------------------------------------------
  NFR-S1   Backend must be stateless
  NFR-S2   Infrastructure must support horizontal scaling
  NFR-S3   Database must allow vertical scaling

## 4.3 Infrastructure Implication

Horizontal Scaling Model:

``` mermaid
flowchart LR
    Users --> LB
    LB --> App1
    LB --> App2
    LB --> App3
```

Scaling must occur without redesign.

# 5. Security Requirements

## 5.1 Business Expectation

-   Protect patient confidentiality
-   Prevent cross-clinic data leakage
-   Enforce strict access control

## 5.2 Formalized Security NFRs

  ID         Requirement
  ---------- ------------------------------------------
  NFR-Sec1   All traffic must use TLS 1.2+
  NFR-Sec2   Database must not be publicly accessible
  NFR-Sec3   Data must be encrypted at rest
  NFR-Sec4   Secrets must not be hardcoded
  NFR-Sec5   Audit logs must be immutable

## 5.3 Infrastructure Implication

Secure Architecture:

``` mermaid
flowchart TD
    Internet --> HTTPS
    HTTPS --> AppService
    AppService --> PrivateEndpoint
    PrivateEndpoint --> Database
```

Database is accessible only through private network paths.

------------------------------------------------------------------------

# 6. Backup & Disaster Recovery Requirements

## 6.1 Business Expectation

-   Daily backups
-   Recovery within 4 hours
-   Maximum 24 hours data loss

## 6.2 Formalized DR NFRs

  ID        Requirement
  --------- ------------------------------------
  NFR-DR1   Automated daily full backups
  NFR-DR2   Transaction log backups enabled
  NFR-DR3   RPO ≤ 24 hours
  NFR-DR4   RTO ≤ 4 hours
  NFR-DR5   Quarterly restore testing required

## 6.3 Backup Model

``` mermaid
flowchart TD
    ProductionDB --> DailyBackup
    ProductionDB --> TransactionLogs
```

Regular restore testing validates recoverability.

------------------------------------------------------------------------

# 7. Observability Requirements

## 7.1 Business Expectation

System must:

-   Detect downtime
-   Detect suspicious login activity
-   Detect performance degradation

## 7.2 Formalized Observability NFRs

  ID       Requirement
  -------- -------------------------------------
  NFR-O1   Centralized log aggregation
  NFR-O2   Alert if error rate \> 5%
  NFR-O3   Alert if latency \> 2 sec sustained
  NFR-O4   Monitor failed login attempts
  NFR-O5   Log retention ≥ 90 days

## 7.3 Observability Model

``` mermaid
flowchart LR
    AppService --> Logs
    AppService --> Metrics
    Logs --> AlertEngine
    Metrics --> AlertEngine
```

Observability ensures fast detection and reduced MTTR.
