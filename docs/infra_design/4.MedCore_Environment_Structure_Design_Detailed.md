# MedCore Infrastructure Design

# Environment Structure Design (Detailed Version)

Document Version: 1.0\
Purpose: Define how Development, Testing, and Production environments
are structured, isolated, and governed within the cloud infrastructure.

------------------------------------------------------------------------

# 1. Introduction

Environment Structure Design defines how we separate different stages of
the software lifecycle in order to protect production stability, ensure
security, control cost, and enable safe development practices.

For MedCore, which handles sensitive patient data, environment
separation is not optional --- it is foundational to security and
compliance.

------------------------------------------------------------------------

# 2. Why Environment Separation Is Required

Without environment separation:

-   Developers could accidentally modify live patient data.
-   Testing bugs could impact real clinics.
-   Security controls might not be validated properly.
-   Production stability could be compromised.
-   Compliance requirements could be violated.

Environment separation reduces operational risk and limits blast radius.

------------------------------------------------------------------------

# 3. High-Level Environment Flow

Changes move progressively from Development to Production.

``` mermaid
flowchart LR
    Developer --> Dev
    Dev --> Test
    Test --> Production
```

No direct deployment from Developer to Production is allowed.

------------------------------------------------------------------------

# 4. The Three Core Environments

## 4.1 Development (Dev)

Purpose: - Feature development - Bug fixes - Experimentation

Characteristics: - Fake or anonymized data - Lower-cost infrastructure -
Flexible configuration - May be unstable

Dev is a safe experimentation zone.

------------------------------------------------------------------------

## 4.2 Testing (Test / QA)

Purpose: - Validate functionality - Validate security controls -
Validate performance under expected load

Characteristics: - Controlled test datasets - Similar configuration to
Production - Moderate stability requirement

Test ensures readiness before go-live.

------------------------------------------------------------------------

## 4.3 Production (Prod)

Purpose: - Live clinics use system - Real patient data stored - Strict
SLA enforcement

Characteristics: - High availability configuration - Full monitoring &
alerting - Strict RBAC - Backup and DR enabled

Production must be stable and secure.

------------------------------------------------------------------------

# 5. Logical Isolation Model

Each environment has independent resources.

``` mermaid
flowchart TD
    subgraph Development
        DevApp
        DevDatabase
    end

    subgraph Testing
        TestApp
        TestDatabase
    end

    subgraph Production
        ProdApp1
        ProdApp2
        ProdDatabase
    end
```

No database or compute resource is shared across environments.

------------------------------------------------------------------------

# 6. Network Isolation Concept

Production network is isolated from Dev and Test networks.

``` mermaid
flowchart TD
    Internet --> DevNetwork
    Internet --> TestNetwork
    Internet --> ProdNetwork

    DevNetwork -. isolated .-> ProdNetwork
    TestNetwork -. isolated .-> ProdNetwork
```

Inter-environment communication is not allowed by default.

------------------------------------------------------------------------

# 7. Access Control Differences

Access is stricter in Production.

  Role        Dev    Test      Production
  ----------- ------ --------- --------------
  Developer   Full   Limited   No DB Access
  QA          No     Full      No
  DevOps      Yes    Yes       Controlled
  DBA         Yes    Yes       Controlled

Principle applied: Least Privilege.

------------------------------------------------------------------------

# 8. Configuration Isolation

Each environment has:

-   Separate secrets
-   Separate connection strings
-   Separate scaling rules
-   Separate monitoring thresholds

``` mermaid
flowchart LR
    DevApp --> DevSecrets
    TestApp --> TestSecrets
    ProdApp --> ProdSecrets
```

Production secrets must never be reused in Dev or Test.

------------------------------------------------------------------------

# 9. Deployment Promotion Model

Deployment must follow promotion flow.

``` mermaid
flowchart LR
    CodeCommit --> DevDeploy
    DevDeploy --> TestDeploy
    TestDeploy --> ProdApproval
    ProdApproval --> ProductionDeploy
```

Manual approval is required before production deployment.

------------------------------------------------------------------------

# 10. Scaling Differences by Environment

Production requires redundancy.

  Setting             Dev        Test       Production
  ------------------- ---------- ---------- ------------
  Min Instances       1          1          2
  Autoscaling         Optional   Optional   Enabled
  Backup Retention    Basic      Moderate   Full
  Monitoring Alerts   Basic      Moderate   Full

Production meets 99.9% availability target.

------------------------------------------------------------------------

# 11. Cost Optimization Strategy

To maintain budget discipline:

-   Dev uses smaller compute tiers
-   Test uses moderate tier
-   Production uses SLA-backed tier

Environment tagging:

-   environment=dev
-   environment=test
-   environment=prod

This enables cost tracking and governance.

------------------------------------------------------------------------

# 12. Risk Mitigation Table

  Risk                            Mitigation
  ------------------------------- --------------------------
  Dev accidentally affects Prod   Separate resource groups
  Prod data leaked to Dev         Data masking policy
  Config drift                    Infrastructure as Code
  Unauthorized Prod access        Strict RBAC + MFA

------------------------------------------------------------------------

# 13. Environment Governance Summary

Environment Structure ensures:

-   Deployment discipline
-   Security isolation
-   Compliance readiness
-   Reduced operational risk
-   Predictable cost management

Environment separation is the first physical control layer in
Infrastructure Design.
