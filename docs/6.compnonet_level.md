# MedCore Secure Patient System

## Component-Level Design (Detailed, Patient Use Case Driven with Mermaid Diagrams)

------------------------------------------------------------------------

# 1. Why Component-Level Design Matters

High-level architecture shows major building blocks like:

-   Frontend
-   Backend
-   Database
-   Identity Provider

But component-level design goes deeper.

It explains:

-   What modules exist inside the backend?
-   How do they interact?
-   Where is validation done?
-   Where is security enforced?
-   Where does audit logging happen?
-   How is tenant isolation guaranteed?

Without component-level clarity:

-   Responsibilities become mixed.
-   Security logic may leak into UI.
-   Business logic may mix with infrastructure logic.
-   Maintenance becomes difficult.
-   Bugs increase.

Component design ensures separation of concerns, maintainability,
security, and clarity.

------------------------------------------------------------------------

Let us walk through a real example.

What happens inside the system?

# 3. Backend Component Overview

Main backend components:

-   API Controller Layer
-   Authentication Middleware
-   Authorization (RBAC) Middleware
-   Tenant Isolation Guard
-   Service Layer (Business Logic)
-   Repository/Data Access Layer
-   Audit Logging Component
-   Exception Handling Component

High-Level Component Flow:

``` mermaid
flowchart TD
    Request --> AuthMiddleware
    AuthMiddleware --> RBACMiddleware
    RBACMiddleware --> TenantGuard
    TenantGuard --> Controller
    Controller --> ServiceLayer
    ServiceLayer --> Repository
    ServiceLayer --> AuditLogger
    Repository --> Database
```

Each component has a clear responsibility.

------------------------------------------------------------------------

# 4. API Controller Layer

Controllers handle HTTP requests.

Responsibilities:

-   Accept request
-   Validate input
-   Call service layer
-   Return response

Important:

Controllers do NOT:

-   Access database directly
-   Implement business logic
-   Implement authorization logic

This separation ensures maintainability.

------------------------------------------------------------------------

# 5. Authentication Middleware

This component:

-   Validates JWT token
-   Verifies signature
-   Validates issuer
-   Validates expiration
-   Extracts claims (user_id, role, clinic_id)

If token invalid:

Request is rejected immediately.

Diagram:

``` mermaid
flowchart LR
    Request --> JWTValidation
    JWTValidation --> ValidUser
    JWTValidation -. reject .-> Unauthorized
```

This protects the system from unauthorized access.

------------------------------------------------------------------------

# 6. Authorization (RBAC) Middleware

After authentication succeeds, authorization checks role permissions.

Example:

Doctor role → Can update patient. Receptionist → Cannot update
diagnosis.

Diagram:

``` mermaid
flowchart LR
    ValidUser --> RBACCheck
    RBACCheck --> Allowed
    RBACCheck -. deny .-> Forbidden
```

Centralized RBAC ensures:

-   No permission logic in controllers
-   No scattered security rules
-   Easy auditing of permission matrix

------------------------------------------------------------------------

# 7. Tenant Isolation Guard

This is critical for multi-clinic support.

TenantGuard ensures:

-   Token clinic_id matches request clinic context
-   All database queries include clinic_id filter

Without this layer:

Cross-clinic data leakage is possible.

Diagram:

``` mermaid
flowchart LR
    RBACSuccess --> TenantValidation
    TenantValidation --> SafeOperation
    TenantValidation -. mismatch .-> AccessDenied
```

This enforces strict data isolation.

------------------------------------------------------------------------

# 8. Service Layer (Business Logic)

Service layer contains domain logic.

For Update Diagnosis:

Steps:

1.  Validate patient exists.
2.  Check not soft-deleted.
3.  Update diagnosis.
4.  Record timestamp.
5.  Trigger audit logging.

Diagram:

``` mermaid
sequenceDiagram
    participant Controller
    participant Service
    participant Repository
    participant Audit

    Controller->>Service: UpdateDiagnosis()
    Service->>Repository: FetchPatient()
    Service->>Repository: SaveChanges()
    Service->>Audit: LogUpdate()
```

Service layer ensures:

-   Business rules centralized
-   Validation consistent
-   Transactions managed correctly

------------------------------------------------------------------------

# 9. Repository Layer (Data Access)

Repository layer:

-   Executes parameterized SQL queries
-   Handles database transactions
-   Applies clinic_id filter
-   Prevents SQL injection

Diagram:

``` mermaid
flowchart LR
    Service --> Repository
    Repository --> Database
```

Repository does not contain business logic.

It only handles data operations.

------------------------------------------------------------------------

# 10. Audit Logging Component

Audit logger records:

-   Who performed action
-   What action
-   On which record
-   At what time
-   From which IP

Write-only pattern.

Diagram:

``` mermaid
flowchart LR
    Service --> AuditLogger
    AuditLogger --> AuditTable
```

Audit is part of transaction to ensure integrity.

------------------------------------------------------------------------

# 11. Exception Handling Component

Centralized exception handler:

-   Captures unhandled exceptions
-   Logs error
-   Returns sanitized error message
-   Prevents sensitive stack trace exposure

This ensures:

-   Security
-   Clean API responses
-   Better debugging

------------------------------------------------------------------------

# 13. Why Separation of Concerns is Important

If we mix everything:

-   Controller handles DB
-   RBAC inside controller
-   Audit inside DB code
-   Tenant filter forgotten in one query

Result:

-   Security bugs
-   Hard-to-maintain code
-   Inconsistent behavior

Clear component boundaries reduce risk.

------------------------------------------------------------------------

# 14. Scalability at Component Level

Because services are stateless:

-   Multiple instances can run.
-   No shared memory dependency.
-   Horizontal scaling possible.

Because repository uses parameterized queries:

-   DB load predictable.
-   Index usage optimized.

Because middleware centralizes checks:

-   Performance overhead minimal.
-   Security consistent.
